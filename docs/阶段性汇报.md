## 一、课题简介

本课题旨在构建一个基于 **LLM-as-Judge** 范式的智能评估智能体，核心功能是对"数据治理前原始文本"与"数据治理后重写文本"进行自动化质量审计。相比传统人工审阅方式，本系统能够在秒级完成多维度评分，并输出带行级定位的结构化瑕疵报告。

系统采用 **DeepSeek 大语言模型**作为裁判引擎，通过精心设计的 System Prompt 约束模型输出，结合 LangChain 框架管理对话链路，最终以 FastAPI + Streamlit 提供可交互的 Web 演示界面。

- 制定人工标注规范：`overall_score`（0~1 浮点）+ `flaws`（瑕疵列表）+ `severity` 三级分类

### 3.2 Day 2（6/29）—— Prompt 调优与多轮验证

**目标：** 针对 Day 1 发现的三类偏差，定向修改 Prompt 并验证效果。

**调优方向：**

1. **数值敏感性强化**：在 `mis_edit` 类型说明中增加量化约束——"数量级变化（如 120万→12万）必须标记为 critical 级瑕疵"
2. **格式优化豁免**：增加豁免条款——"纯标点/空格/段落对齐调整不应扣除语义一致性分"
3. **过度清洗权重提升**：增加量化阈值——"删除原文超过 30% 内容应判定为 critical 级结构性瑕疵"
4. **"合理改写"与"错误改写"边界细化**：补充边界定义，提升相似样本判断一致性

**验证过程：**

- 第一轮调优后重新校准，Pearson r 从 0.61 提升至约 0.72
- 第二轮针对格式优化误罚，高偏差样本从 3 条减少至 1 条，r 提升至约 0.78
- 第三轮针对过度清洗识别不足，r 达到 0.8 以上

**关键产出：** 优化后的评分 Prompt + 多轮校准验证记录 + 高偏差样本问题分类

**Phase 2 — 抗偏置与稳定性（6 项全部完成）：**

| 任务 | 模块 | 说明 |
|------|------|------|
| P2-1 长度偏置检测 | `app/debias.py` | 治理前后文本长度比，High/Medium/Low 三级风险 |
| P2-2 位置偏置检测 | `app/debias.py` | 瑕疵分布是否集中在前/后半部分（>70% = 偏置） |

## 六、技术亮点

1. **三层 JSON 兜底解析**：直接解析 → Markdown 围栏提取 → 正则花括号提取，应对 DeepSeek 偶发的 `<think>` 标签输出
2. **规则引擎 + LLM 双通道**：确定性规则检测数值变化和结构丢失，LLM 负责语义层面判断，互为补充
3. **临界区自动重采样**：评分落在 0.75-0.85 时自动多跑 2 次取平均，减少边界误判
4. **数据驱动的 Prompt 调优**：不是拍脑袋修改，而是"发现偏差→归因→定向修改→校准验证"的迭代循环
5. **Profile 差异化评估**：针对政府公告、法律文书等不同文本类型，动态调整关键事实敏感度和扣分策略

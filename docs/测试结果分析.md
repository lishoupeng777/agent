## 二、6 个 GAP 样本逐条分析

### 2.1 sample_005 — 表格结构破坏（diff=0.186，LLM 偏低）

| | LLM | Human |
|---|---|---|
| 总分 | 0.649 | 0.835 |
| 判定 | review | — |

**原文：** Markdown 表格（产品/价格/销量），治理后转为纯文本叙述。

**GT 瑕疵：** 1 条 critical 级 structure — 表格结构被完全破坏。

**Human 四维度评分：**

| 维度 | 分数 | 说明 |
|------|------|------|
| 语义一致性 | 0.90 | 数据全部保留 |
| 过度清洗 | 0.80 | 未删除内容，仅转换格式 |
| 可读性结构 | 0.45 | 表格结构丧失 |
| 事实准确性 | 0.95 | 数值无误 |

**偏差原因：** 人工认为"数据保真度高（0.9/0.95），只是结构差（0.45）"，加权后总分仍较高。但 LLM 可能将结构破坏视为更严重的整体问题，对语义和事实维度也产生了连带扣分。

**结论：LLM 对结构瑕疵的"传染效应"过强——一个维度的严重瑕疵拉低了其他维度的分数。**

### 2.2 sample_010 — 机构名误改（diff=0.180，LLM 偏低）

**GT 瑕疵：** 2 条（1 条 major 机构名缩写 + 1 条 critical 技术参数误改）。

**Human 四维度评分：**

| 维度 | 分数 | 说明 |

**结论：LLM 的软惩罚机制对多条瑕疵的叠加效应比人工更敏感。人工是"分维度独立评判"，LLM 倾向于"整体印象评判"。**

### 2.3 sample_013 — 列表结构破坏（diff=0.433，最大偏差）

| | LLM | Human |
|---|---|---|
| 总分 | 0.267 | 0.700 |
| 判定 | fail | — |

**原文：** 5 项编号列表（含括号补充说明），治理后压缩为一行逗号连接文本。

**GT 瑕疵：** 1 条 major 级 structure — 列表结构丢失。

**Human 四维度评分：**

| 维度 | 分数 | 说明 |
|------|------|------|
| 语义一致性 | 0.75 | 材料名称全部保留 |
| 过度清洗 | 0.55 | 编号和括号说明丢失 |
| 可读性结构 | 0.50 | 列表变为单行 |
| 事实准确性 | 0.80 | 内容无误 |

**偏差原因：** 这是最大偏差样本。人工认为"内容基本保留（语义 0.75，事实 0.80），只是结构变差（0.50）"，加权后仍得 0.70。但 LLM 给出 0.267，接近 fail 底部。

**深层原因分析：**
1. **规则引擎过度惩罚**：`detect_flaws()` 可能将列表→文本的转换检测为多条瑕疵（每个列表项各一条），而非一条结构瑕疵
2. **LLM 将结构问题泛化**：列表结构丢失被 LLM 解读为"信息丢失"，连带拉低语义和事实分数
3. **人工与 LLM 对"结构"的理解差异**：人工认为"材料清单还在，只是格式变了"；LLM 认为"结构化信息被破坏 = 信息损失"

**结论：这是典型的"LLM 对结构瑕疵过度惩罚"问题。需要在 Prompt 中明确区分"结构变化"和"内容丢失"。**

### 2.4 sample_022 — 数值误改（diff=0.329，LLM 偏低）

| | LLM | Human |
|---|---|---|
| 总分 | 0.306 | 0.635 |
| 判定 | fail | — |

**原文：** 营业收入从 126.8 亿元改为 128.6 亿元（变化 1.4%）。

**GT 瑕疵：** 1 条 critical 级 mis_edit — 营业收入数值被篡改。

**Human 四维度评分：**

| 维度 | 分数 | 说明 |
|------|------|------|
| 语义一致性 | 0.78 | 仅一个数值变化 |
| 过度清洗 | 0.95 | 未删减内容 |
| 可读性结构 | 1.00 | 结构完整 |
| 事实准确性 | 0.20 | 关键财务数据错误 |

**偏差原因：** 人工认为"只有一个数错了（事实 0.20），其他都很好（0.78/0.95/1.00）"，加权后得 0.635。但 LLM 给出 0.306，几乎是人工分数的一半。

**深层原因分析：**
1. **LLM 的"一票否决"倾向**：发现一个 critical 级数值错误后，LLM 可能将其视为"整体不可信"，连带拉低所有维度
2. **Prompt 中的惩罚因子过强**：`apply_soft_penalty()` 中 critical 事实瑕疵的惩罚因子是 0.6，乘以原始分后大幅拉低
3. **人工的"容错思维"**：人工认为"一个数错了但其他信息仍有价值"，LLM 缺乏这种容错判断

**结论：LLM 对单条 critical 瑕疵的惩罚力度比人工更强。需要调整软惩罚因子，或在 Prompt 中增加"单条瑕疵不应导致整体分数过低"的指引。**

### 2.5 sample_024 — 过度清洗（diff=0.352，LLM 偏低）

| | LLM | Human |
|---|---|---|
| 总分 | 0.425 | 0.777 |
| 判定 | fail | — |

**原文：** "人工智能、大数据、工业互联网及数字孪生四个方向"→"人工智能和大数据"（删除了两个方向）。

**GT 瑕疵：** 1 条 major 级 over_clean — 删除了"工业互联网"和"数字孪生"。

**Human 四维度评分：**

| 维度 | 分数 | 说明 |
|------|------|------|
| 语义一致性 | 0.72 | 两个方向被删除 |
| 过度清洗 | 0.45 | 明确的过度治理 |
| 可读性结构 | 1.00 | 结构完整 |
| 事实准确性 | 0.88 | 保留的内容无误 |

**偏差原因：** 人工认为"删除了两个方向是过度清洗（0.45），但保留的内容都正确（0.88），结构没坏（1.00）"，加权后得 0.777。LLM 给出 0.425，差距 0.352。

**深层原因分析：**
1. **LLM 将 over_clean 严重程度拔高**：GT 标注为 major，但 LLM 可能判定为 critical（删除业务方向 = 重要信息丢失）
2. **规则引擎的数值敏感性干扰**：四个方向删了两个 = 50% 删除率，规则引擎可能将其标记为 critical（超 30% 阈值）
3. **人工的"比例判断"**：人工认为"删了两个方向，但原文也就四个，不算特别严重"；LLM 更倾向于是非判断

**结论：规则引擎的"删除超 30% = critical"阈值在此场景下过于机械。四个方向删两个确实是 50%，但人工认为严重程度是 major 而非 critical。**

## 四、F1 偏低原因分析

| 指标 | 数值 | 说明 |
|------|------|------|
| TP | 13 | LLM 检出且 GT 有的瑕疵 |
| FP | 17 | LLM 检出但 GT 没有的瑕疵（误报） |
| FN | 5 | GT 有但 LLM 没检出的瑕疵（漏报） |
| Precision | 0.4333 | 13 / (13+17) |
| Recall | 0.7222 | 13 / (13+5) |

**核心问题：FP=17 > TP=13，误报比正确检出还多。**

可能原因：

1. **规则引擎过度检测**：`detect_flaws()` 的阈值设置过于敏感，将正常表述差异也标记为瑕疵
2. **LLM 输出了 GT 未标注的瑕疵类别**：GT 可能只标注了主要瑕疵，LLM 检出了更多细节差异
3. **瑕疵粒度不匹配**：LLM 可能将一条大瑕疵拆分为多条小瑕疵（如将"多处数值变化"拆为每处各一条）
4. **类别映射问题**：LLM 输出的 category 与 GT 的 category 不完全一致，导致匹配失败

**改进方向：**
- 调整 `detect_flaws()` 的检测阈值（如数值变化从 10% 提高到 15%）
- 在 Prompt 中限制瑕疵粒度，要求"合并同类瑕疵"
- 检查 LLM 输出的 category 与 GT 的映射关系

## 五、锚点准确率偏低原因分析

| 指标 | 数值 | 说明 |
|------|------|------|
| anchor_accuracy | 0.7222 | 字符位置匹配（容差 10） |
| snippet_accuracy | 0.3333 | 原文片段匹配 |

**核心问题：snippet_accuracy 只有 0.33，LLM 输出的 snippet 与 GT 标注的原文片段不匹配。**

可能原因：

1. **LLM 改写了 snippet**：LLM 不是直接摘录原文，而是用自己的话描述位置（如输出"表格被转为文本"而非原文"A产品99元售出1200件"）
2. **GT 的 snippet 格式不统一**：部分 GT 的 snippet 是原文摘录，部分是描述性文字
3. **segment_id 不匹配**：LLM 输出的 segment_id 与 GT 的不一致

**改进方向：**
- 在 Prompt 中明确要求"snippet 必须是原文中的精确文本片段"
- 检查 GT 的 snippet 格式是否统一
- 使用更宽松的匹配策略（如子串包含而非精确匹配）

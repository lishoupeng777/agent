# 评估模式（Evaluation Profile）扩展设计

## 目标

将当前固定的一套内容保真度评估逻辑，扩展为可选择、可扩展的评估模式体系。首要目标是解决政务通告类文本在关键事实删减时仍被判高分通过的问题，同时保留通用文本场景下的现有可用性。

本次设计不把产品固化为“政务/通用二选一”，而是引入统一的 profile 框架，允许后续继续增加更多面向特定文本类型的严格模式。

## 问题定位

当前高分误判的根因主要有三点：

1. `app/prompts.py` 的 system prompt 虽然强调金额、百分比、日期等误改属于严重问题，但没有明确把“关键事实被删除、泛化、区间被抹平、期限被删除”定义成足够强的惩罚条件。
2. `app/engine.py` 的后处理 veto 只依赖 LLM 返回的 `flaws` 类别与严重程度，不会主动检查关键事实缺失。因此只要模型没有把删失标成 `critical`，总体分仍可能维持较高。
3. 前端和 API 当前没有“评估模式”概念，导致所有文本共享同一套容忍度，无法对政务、法规、合同等高保真文本类型使用更严格规则。

## 设计原则

1. **单引擎，多 profile**：保留一套 `evaluate()` 主流程，不分叉成两套独立引擎。
2. **基础规则 + profile 补充规则**：通用规则继续存在，不同模式只叠加差异化约束。
3. **LLM 判断 + Python 兜底**：让 LLM 负责语义判断，同时在高风险事实上增加确定性的后处理检查。
4. **默认行为稳定**：默认 profile 仍为 `general`，避免现有调用方在未显式选择时行为突变。
5. **面向扩展设计**：当前只内置少量 profile，但数据模型、注册表、前端选项、结果展示都按未来继续扩展来设计。

## 首批内置 profile

### 1. `general`
适用于普通说明文、常规治理文本、一般内容清洗场景。

特点：
- 保持现有四维度评分逻辑为主
- 对明显误改、严重结构破坏继续严惩
- 对删减和泛化只适度收紧，避免误杀大量普通文本

### 2. `government_notice_strict`
适用于政务通告、公告、制度通知、执法告知等文本。

重点关注以下关键事实是否被删除、泛化或区间抹平：
- 生效日期、失效日期、截止日期、有效期
- 办理时限、报告时限、响应时限
- 金额、罚款区间、收费标准
- 百分比、数量门槛、面积/重量/身高等阈值
- 适用范围、区域范围、对象范围
- 豁免对象、例外对象
- 处罚措施、责任后果

这类文本中，上述信息不是装饰性细节，而是内容保真的核心组成部分。

### 3. `legal_strict`
适用于法规、合同、条款、协议、制度规范等文本。

额外重点关注：
- 责任主体
- 义务与禁止性规定
- 条件触发
- 例外条款
- 法律后果或违约后果

对于这类文本，若改写导致义务边界或责任后果被弱化，应直接重罚。

## 数据模型变更

在 `python_version/app/models.py` 的 `EvalRequest` 中新增字段：

- `evaluation_profile: str = "general"`

设计要求：
- 字段对 API、Streamlit UI、静态 HTML 前端统一可用
- 默认值必须为 `general`
- 暂不强制使用枚举类型，先使用字符串配合后端注册表校验，以便后续平滑扩展
- 对未知 profile，后端回退到 `general` 或返回清晰错误，二选一必须统一；推荐返回清晰错误，避免静默误判

在 `EvalResponse` 中新增字段：
- `evaluation_profile: str`

用于结果页展示“本次评估使用了哪种规则集”。

## Profile 注册表设计

新增一个集中式 profile 定义模块，建议放在 `python_version/app/` 下，例如：
- `profiles.py`

该模块负责维护每个 profile 的配置，至少包含：

1. `key`：profile 标识，如 `general`
2. `label`：前端展示名，如“通用文本”
3. `description`：简短说明
4. `prompt_rules`：注入到 system prompt 的补充规则
5. `critical_fact_types`：该 profile 关注的高风险事实类别
6. `penalty_policy`：后处理降分/封顶规则

这样 `prompts.py` 与 `engine.py` 不需要写死 if/else 到处散落，而是通过 profile 配置读取规则。

## Prompt 层设计

`python_version/app/prompts.py` 调整为：

- 基础 system prompt 继续保留四维度和通用输出格式要求
- 根据 `evaluation_profile` 叠加 profile-specific 的补充规则与 few-shot 示例

### `general` 的 prompt 方向
- 延续现有四维度定义
- 明确“重要事实删失也会扣分”，但保持整体平衡

### `government_notice_strict` 的 prompt 方向
要明确告诉模型：
- 对政务通告，数字、日期、罚则、期限、适用范围、例外对象属于关键事实
- “保留大意但删除关键细节”不能算高保真
- 将“罚款 2000~5000 元”改成“处罚款”属于显著信息损失
- 将“有效期至某日”删除属于关键信息删失
- 将“肩高 61 厘米或体重 30 公斤”改成“大型犬”属于阈值丢失

### `legal_strict` 的 prompt 方向
要强调：
- 责任主体、义务、禁止、条件、例外、法律后果不能模糊化
- “维持大意”不足以视为高保真
- 删除条件限制或责任后果时，要优先标为 `mis_edit` 或 `over_clean`

### Few-shot 调整
新增与本项目高风险场景更接近的 few-shot：
- 日期和有效期被删除
- 罚款区间被泛化
- 数量阈值被删除
- 政务通告被压缩成“原则性表述”
- 法律条款中责任主体或后果被弱化

## 引擎后处理设计

`python_version/app/engine.py` 需要从“只看 flaws 的严重程度”升级为“profile-aware 后处理”。

### 保留现有通用 veto
对于 `general`：
- `critical structure` 或 `critical over_clean` → 总分 ≤ 0.25
- `critical mis_edit` → 总分 ≤ 0.35

### 新增 profile-aware 关键事实检查
在 LLM 输出解析之后，增加一层 Python 后处理检查：

1. 从 `before_text` 中提取高风险事实线索
2. 检查 `after_text` 是否仍保留对应事实或等价表达
3. 若明显缺失，则追加 flaw，并执行额外降分/封顶

建议先做轻量规则，不做复杂 NLP：
- 数字：阿拉伯数字、中文数字、百分号
- 日期：`2024年6月1日`、`2027年5月31日` 等
- 区间：`2000元以上5000元以下`、`5日以上10日以下`
- 时限：`15日内`、`24小时内`
- 阈值：`肩高超过61厘米`、`体重超过30公斤`

### `government_notice_strict` 的后处理策略
建议采用分层惩罚，而不是只靠 `critical`：

- 发现 1 个关键事实严重缺失：
  - 至少追加 `major` flaw
  - 总体判定不能高于 `review`
- 发现多个关键事实缺失，或涉及处罚区间/有效期/适用范围等核心事实：
  - 追加 `critical` flaw
  - 总分封顶到 `0.35` 或更低
  - 倾向判 `fail`
- 若出现“原文有具体数字，改写后只剩抽象说法”：
  - 追加 `over_clean` 或 `mis_edit`
- 若处罚措施仍在，但区间删除：
  - 不能视为 minor，至少 `major`

### `legal_strict` 的后处理策略
- 删除责任主体、义务、条件、例外、法律后果：至少 `major`
- 若导致责任边界变化或法律后果弱化：`critical`
- 该类问题至少压到 `review`，严重时直接 `fail`

### 实现边界
本轮调优不追求“自动完整理解所有法律逻辑”，而是先抓最容易稳定识别、最容易造成误判的显式事实。保持规则清晰、可解释、可维护。

## API 变更

### FastAPI
`python_version/app/routes.py` 中的 `/evaluate`、`/stability`、`/calibrate`、`/batch/evaluate` 等接口都应支持新的 `evaluation_profile` 字段。

要求：
- 单条评估显式透传 profile
- 稳定性分析沿用同一 profile
- 校准与批量评估允许按请求记录分别带 profile
- 若批量中未传，使用默认 `general`

### 响应
结果中返回 `evaluation_profile`，便于前端展示、历史记录复现与调试。

## Streamlit UI 设计

在 `python_version/app.py` 的侧边栏中新增“评估模式”选择。

展示文案建议：
- 通用文本
- 政务通告/公告（严格保真）
- 法规合同/条款（严格保真）

交互要求：
- 默认选中“通用文本”
- 评估请求中把选择结果写入 `EvalRequest`
- 结果区展示“本次模式”
- 批量评估第一阶段先统一使用侧边栏所选 profile，不额外为 CSV 每行开放不同 profile，以控制界面复杂度

## 静态 HTML 前端设计

在 `python_version/static/index.html` 的“核心评估”区域新增 profile 下拉选择。

要求：
- 与 Streamlit 使用相同的 profile 集合与命名
- 提交 `/evaluate` 时把 `evaluation_profile` 一并发给后端
- 结果卡片展示“评估模式”
- “使用说明”页补充 profile 说明，解释何时应选择严格模式

稳定性分析页也应支持 profile 选择，避免稳定性分析与核心评估使用不同规则。

## 历史记录与可复现性

由于 profile 会影响评分与 prompt 内容，因此以下位置都要纳入 profile：

- `reproducibility_token` 的生成输入
- `prompt_version` 的计算语义
- 历史持久化记录
- 缓存键（如果当前缓存依赖输入文本和模型，也要补上 profile）

否则会出现“同样文本在不同模式下分数不同，但被错误复用旧结果”的问题。

## 对当前问题样本的预期行为

对于用户提供的“养犬管理通告”样本：

### 在 `general` 模式下
预期从当前的 `pass` 降到 `review` 附近，原因应明确指出：
- 删除投诉量增长 `42.3%`
- 删除大型犬判定阈值
- 删除罚款区间
- 删除有效期截止日

### 在 `government_notice_strict` 模式下
预期明显降分，倾向 `review` 或 `fail`，且至少给出以下一种明确判定：
- 关键期限丢失
- 罚则区间被泛化
- 阈值标准被删除
- 适用规则精度下降

如果依然给出高分通过，则视为本轮调优失败。

## 测试与验证

本项目没有 pytest 自动化测试套件，因此本轮验证应包含两类：

### 1. 代码层验证
新增少量可离线运行的纯 Python 单元测试，优先覆盖：
- profile 注册表行为
- 未知 profile 校验
- 关键事实提取/缺失检查函数
- 不同 profile 下的后处理封顶逻辑

这些测试不应依赖外部 API。

### 2. 真实样本验证
对至少以下样本做手工/脚本验证：
- 当前养犬通告样本
- 一个普通文本轻微润色样本（应保持高分）
- 一个法律/条款类删责任后果样本（应显著降分）
- 一个结构改动但信息完整的普通文本样本（避免过度误杀）

## 实施顺序

1. 在数据模型中加入 `evaluation_profile`
2. 引入 profile 注册表与配置结构
3. 改造 prompt 构建逻辑，支持 profile-specific 规则
4. 在 engine 中加入 profile-aware 后处理与关键事实缺失检查
5. 更新 reproducibility token / 缓存键 / 历史记录相关逻辑
6. 更新 Streamlit UI
7. 更新静态 HTML 前端
8. 加入离线测试与样本验证

## 风险与取舍

### 风险 1：严格模式误杀增加
这是预期内取舍。严格模式本来就应优先降低漏判，允许更多 `review`。

### 风险 2：规则膨胀
如果把所有文本类型一次性做全，会迅速失控。因此首批只内置 3 个 profile，并把 profile 注册表做成可扩展结构。

### 风险 3：LLM 与规则冲突
可能出现模型打高分、规则强压低分的情况。对此应接受，因为高风险事实缺失比“模型感觉大意一致”更重要。结果解释里要能体现压分原因。

## 推荐结论

采用“可扩展 profile 框架 + LLM 判断 + Python 关键事实兜底”的方案。

这既能修复当前政务文本高分误判问题，也不会把系统写死成“只有政务/通用两种模式”。后续新增新闻、医疗、合同、制度解释等 profile 时，只需要扩展注册表、补充 prompt 规则和后处理策略，而不需要重写整套评估引擎。

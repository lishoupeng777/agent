# 课题12评估智能体验收修复与简历增强设计

## 目标

将当前“内容保真度与治理质量评估智能体”从可运行原型提升为适合课题验收与简历展示的研究原型增强版：一方面修复当前评测中未达标或不可信的指标（Pearson、一致性口径、稳定性覆盖、异常样本解析）；另一方面保留并增强其可解释、可复现、可追踪的工程表达，使其能够作为“大厂简历加分项目”讲清楚研究问题、系统实现与实验闭环。

## 范围

本轮只覆盖两层工作：

1. **验收修复层**
   - 修复 `sample_009` 一类“原始模型输出正常，但最终结果退化”的解析/后处理异常。
   - 统一锚点定位验收口径，明确报告中的达标判断与展示指标一致。
   - 让稳定性分析在测试脚本与综合报告中真实执行并可验证，不再出现 `0/0` 空汇总。
   - 调整总分聚合与惩罚逻辑，使结构破坏、严重可读性下降、关键事实删改对总分有足够压制，从而提升与人工评分的一致性。

2. **研究原型增强层**
   - 强化评测报告中的“研究叙事一致性”：指标定义、达标条件、样本覆盖范围、可复现实验信息应一致。
   - 强化项目的“简历表达素材”：稳定性、一致性、锚点定位、偏置分析、可复现令牌等能力在代码与报告里形成闭环。
   - 仅做轻量工程增强，不引入超出课题要求的大规模平台化改造。

## 不做的事

- 不做正式生产级平台化建设（多租户、鉴权体系、消息队列、云部署编排）。
- 不做与课题主线无关的大范围重构。
- 不私自扩大功能边界到“通用生成评测平台”。

## 现状问题

### 1. 稳定性报告口径失真
当前测试脚本调用综合报告时明确关闭了稳定性分析，但报告摘要仍打印稳定率与稳定/总数，导致本次测试中显示 `0/0`，容易被误读为“稳定性不达标”，而真实情况是“本次未测”。

### 2. 锚点定位达标判断与展示值不一致
当前控制台向用户展示 `location_f1`，但达标判断使用 `anchor_accuracy=location_recall`。因此会出现“展示值 0.9143、标签却未达标”的口径冲突，削弱报告可信度。

### 3. 解析/后处理存在异常退化样本
至少 `sample_009` 出现了明显异常：原始 LLM 输出中包含完整维度与 flaws，但最终结果中维度被回退为统一 `0.5`，同时 flaws 为空。此类问题会直接拉低一致性指标，并削弱系统可信度。

### 4. 总分压制不足
对“结构严重破坏”“可读性显著崩坏”“关键要点大幅删减”的样本，虽然局部维度已识别出问题，但整体总分仍可能偏高，与人工评审的风险判断不一致，影响 Pearson。

## 设计方案

### 一、验收修复层

#### A. 解析链路加固
重点排查并修复从 `raw_llm_output` → 解析结果 → 规范化维度名 → flaws 验证/过滤 → `EvalResponse` 生成之间的退化路径。

设计要求：
- 当原始 JSON 已包含合法维度与 flaws 时，不允许静默退回“全 0.5 默认分 + 空 flaws”，除非明确记录触发原因。
- 解析失败、字段缺失、枚举不合法、二阶段 verifier 剔除等不同情形要可区分。
- 对异常退化样本提供结构化 debug 信息，便于后续解释为什么某条样本分数异常。

#### B. 锚点指标口径统一
明确“课题验收中的锚点定位准确率”在本项目中采用哪个指标作为主指标，并让：
- JSON 报告字段
- 控制台摘要
- 达标布尔值
- 综合判定 checklist

全部使用同一口径。

推荐设计：
- 将 `location_recall` 作为“锚点覆盖率”单独展示。
- 将 `location_f1` 作为“锚点定位主指标”用于达标判断与报告强调。
- 若仍需保留 `anchor_accuracy` 字段，则明确其定义，不再让其与“准确率”自然语言描述混淆。

#### C. 稳定性测试真实纳入评测
将测试脚本改为支持明确开启稳定性分析，并在报告中区分：
- 本次是否执行稳定性分析
- 稳定性采样覆盖了哪些样本
- 未运行时显示“未测试”，而不是 `0/0`

推荐对综合评测默认启用有限样本稳定性测试（如抽样 5 条、每条 3 次），控制成本的同时满足课题证据要求。

#### D. 总分与惩罚规则校准
保留现有多维打分框架，但增强高风险场景的总分抑制：
- 表格/列表被拍平成散文且结构对理解重要时，整体分数需显著下压。
- 可读性进入“严重电报体/极端缩写”时，即使语义还在，也不能维持高总分。
- 关键量化指标、法律限定语、时限目标等被删除时，要有更强的整体扣分传导。

目标不是“靠写死规则过样本”，而是让“局部高分 + 局部致命问题”不会被简单平均掩盖。

### 二、研究原型增强层

#### A. 评测报告叙事一致化
报告中每项指标需回答三个问题：
1. 它测什么？
2. 本次是否实际执行？
3. 达标阈值是什么？

这样你在答辩或面试时，可以直接拿报告解释：
- 一致性如何衡量
- 稳定性如何证明
- 锚点定位如何定义
- 可复现如何保证

#### B. 可追踪性增强
对于评估异常样本，应保留足够证据说明：
- 原始模型输出是什么
- 二阶段 verifier 是否剔除了 flaws
- 最终分数经过了哪些规则调整
- 为什么得到当前 verdict

这会让项目更像“可审计的评估系统”，而不仅是“调用大模型后输出一个分数”。

#### C. 简历表达增强
改造完成后，项目可被你概括为：
- 面向“治理前后文本比对”的专项评估智能体
- 支持多维评分、锚点级瑕疵定位、二阶段验证、稳定性分析、一致性校准、偏置检测、可复现实验
- 通过规则校准与评测链路修复，将核心验收指标提升到目标线附近或之上

这类表述比“做了一个 FastAPI + 大模型项目”更有辨识度。

## 涉及模块

- `python_version/app/chain.py`
  - 核心评估、解析、后处理、总分聚合的主链路。
- `python_version/app/verifier.py`
  - 二阶段瑕疵验证逻辑，需确认是否误删了本应保留的 flaws。
- `python_version/app/reporter.py`
  - 综合报告字段、达标判断、控制台摘要、稳定性汇总、指标口径统一。
- `python_version/app/metrics.py`
  - 锚点定位指标定义与返回字段。
- `python_version/tests/test_evaluate.py`
  - 综合评测的执行参数与输出期望。
- `python_version/eval_report.json`
  - 作为回归验证产物，需体现新口径与新结果。

## 验证方式

完成后至少做一轮真实回归：
- 重新运行 `python tests/test_evaluate.py`
- 对比修复前后的：Pearson、锚点主指标、稳定性输出、整体 checklist
- 抽查异常样本（至少 `sample_005`、`sample_007`、`sample_009`）确认修复方向符合预期

## 成功标准

### 课题侧
- Pearson r 达到或超过 0.8，或至少具备明确、可解释、可复现实验依据说明剩余差距。
- 稳定性指标真实输出，不再用空汇总充当结果。
- 锚点定位达标判断与展示口径完全一致。
- 可复现、可解释、锚点定位、瑕疵检出链路能用统一报告说清楚。

### 简历侧
- 项目可以被描述为“有专项研究问题、有指标闭环、有工程实现、有实验报告”的完整系统。
- 代码与报告体现出不仅能调用模型，还能做校准、验证、追踪与可复现控制。

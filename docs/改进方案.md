# 评估系统改进方案

基于课题12"稳、准、可复现、锚点定位"的研究要求，针对当前系统的四个核心痛点提出改进方案。

## 一、锚点定位：用锚点标识符替代字符偏移

### 问题

LLM 不擅长计算字符位置，`start_char=0` 频繁出现，锚点准确率低。

### 现状

系统已有 `build_anchored_text()` 注入 `[Before N]/[After N]` 段落标记，但 LLM 输出时仍被要求给出 `start_char/end_char` 绝对位置，导致"字符偏移幻觉"。

### 改进方案

**Prompt 层**：修改输出格式要求，LLM 只输出锚点编号，不输出字符偏移。

```json
// 改前
"location": {"segment_id": "After 5", "start_char": 0, "end_char": 20, "snippet": "..."}

// 改后
"location": {
  "before_anchor": "[Before 2]",
  "after_anchor": "[After 2]",
  "snippet": "断开电源并静置至少5分钟"
}
```

**后处理层**：用算法将锚点编号映射为字符偏移。

## 二、双向映射：瑕玼同时定位前后文本

### 问题

当前瑕玼只指向治理后文本（`segment_id: "After 5"`），无法追溯到治理前对应位置。

### 改进方案

**Prompt 层**：要求 LLM 同时输出前后锚点。

```json
"mapping": {
  "source_anchor": "[Before 2]",
  "target_anchor": "[After 2]",
  "source_snippet": "断开电源并静置至少5分钟",
  "target_snippet": "断开电源"
}
```

**后处理层**：算法层 `detect_flaws()` 已有段落配对逻辑（`SequenceMatcher`），可自动补充双向映射。

```python
def enrich_flaw_mapping(flaws, anchored_before, anchored_after):
    """为 LLM 输出的瑕玼补充前后锚点映射"""
    for flaw in flaws:
        after_anchor = flaw.get("location", {}).get("after_anchor", "")
        # 在 anchored_before 中找到最相似的段落
        before_anchor = find_matching_anchor(after_anchor, anchored_before)
        flaw["mapping"] = {
            "source_anchor": before_anchor,
            "target_anchor": after_anchor,
        }
```

**预期效果**：支持前后文本并排高亮对比，提升可解释性。

## 三、维度正交：消除语义与事实的重叠扣分

### 问题

当前 `semantic` 和 `factual` 维度对"信息删除"同时扣分，导致维度不独立、方差增大。

### 现状定义

| 维度 | 当前定义 | 问题 |
|------|---------|------|
| semantic (0.35) | 语义是否一致，核心信息是否完整 | "信息删除"扣分 |
| factual (0.35) | 事实是否准确，是否被篡改 | "信息删除"也扣分 |

"删除安全警告"同时被两个维度扣分 → 维度重合 → 评分方差增大。

### 改进方案

重新定义维度边界，使其正交：

| 维度 | 新定义 | 扣分场景 |
|------|--------|---------|
| **semantic_fidelity** (0.35) | 信息是否完整保留（保真度） | 信息被删除、被泛化、被遗漏 |
| **factual_consistency** (0.35) | 事实是否被篡改（一致性） | 数值被改、日期被改、无中生有 |
| **structure_quality** (0.15) | 结构是否保持 | 列表被删、表格被破坏 |
| **readability** (0.15) | 可读性是否合格 | 语句不通、标点错误 |

**核心区别**：
- "删除安全警告" → 只扣 `semantic_fidelity`（信息缺失），不扣 `factual_consistency`
- "把5分钟改成50分钟" → 只扣 `factual_consistency`（事实篡改），不扣 `semantic_fidelity`
- 两者同时发生 → 各自独立扣分

**Prompt 修改**：

```
**【维度评分边界】**
- semantic_fidelity：治理后文本是否保留了原文的完整信息？缺失、遗漏、泛化扣此维度。
  注意：信息被"删除"是保真度问题，不是事实问题。
- factual_consistency：治理后文本是否篡改了原文的事实？数值、日期、名称被改扣此维度。
  注意：信息被"删除"不算事实篡改，只有"改错了"才算。
```

**预期效果**：维度间相关性降低，Pearson r 提升 0.02~0.05，评分更稳定。

## 四、决策路由：增加机器可读的判定原因

### 问题

当前 `verdict` 只有 pass/review/fail 三个值，下游系统无法得知判定原因。

### 改进方案

在评估响应中增加 `reason_code` 和 `reject_reasons`：

```json
{
  "verdict": "fail",
  "reason_code": "CRITICAL_FLAW_DETECTED",
  "reject_reasons": [
    {
      "code": "CONSTRAINT_DELETION",
      "description": "关键约束词'至少5分钟'被删除",
      "severity": "critical",
      "flaw_index": 0
    }
  ]
}
```

**reason_code 枚举**：

| code | 含义 | 触发条件 |
|------|------|---------|
| `PASS_ALL_CLEAR` | 全部通过 | 无瑕玼，分数 >= 0.82 |
| `SCORE_BELOW_THRESHOLD` | 分数不足 | overall_score < 0.5 |
| `CRITICAL_FLAW_DETECTED` | 严重瑕玼 | 存在 critical 级瑕玼 |
| `CONSTRAINT_DELETION` | 约束词删除 | 检测到约束词被删除 |
| `FACTUAL_DISTORTION` | 事实篡改 | 数值/日期被错误修改 |

**实现位置**：`chain.py` 的后处理层，不修改 LLM 输出。

```python
def determine_verdict_with_reason(overall_score, flaws, dimensions):
    reasons = []
    if any(f.severity == "critical" for f in flaws):
        reasons.append({"code": "CRITICAL_FLAW_DETECTED", ...})
    if overall_score < 0.5:
        reasons.append({"code": "SCORE_BELOW_THRESHOLD", ...})
    # ...
    verdict = determine_verdict(overall_score)
    reason_code = reasons[0]["code"] if reasons else "PASS_ALL_CLEAR"
    return verdict, reason_code, reasons
```

**预期效果**：下游质检流水线可根据 `reason_code` 自动路由（如 `CONSTRAINT_DELETION` → 人工复核，`FACTUAL_DISTORTION` → 自动回滚）。

## 五、改进优先级

| 优先级 | 改进项 | 预期收益 | 实现成本 |
|--------|--------|---------|---------|
| P0 | 维度正交（消除重叠扣分） | 评分更稳定，Pearson r +0.02~0.05 | 改 Prompt，约 2h |
| P0 | 锚点标识符替代字符偏移 | 锚点准确率 88.9% → 95%+ | 改 Prompt + 后处理，约 4h |
| P1 | 双向映射 | 可解释性提升，支持前后对比 | 改 Prompt + 后处理，约 3h |
| P2 | 决策路由 reason_code | 下游系统可自动路由 | 改后处理，约 1h |

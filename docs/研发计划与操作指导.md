## 1. 研发总览

### 1.1 项目目标

构建一个 **LLM-as-Judge** 智能体，自动比对"数据治理前"与"数据治理后"的文本对，从以下维度进行质量审计：

| 维度 | 说明 | 权重 |
|---|---|---|
| 语义一致性 | 核心事实、实体、量化指标是否被误改或丢失 | 0.5 |
| 可读性与结构质量 | 排版、语句通顺度、表格对齐 | 0.5 |

### 1.2 核心架构

```
用户输入 (治理前/后文本)
       │
       ▼
┌─────────────────┐
│  锚点预处理模块   │  ← build_anchored_text()
│  [Before N] 标尺  │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  Prompt 构建模块  │  ← System Prompt + User Template
│  裁判规则注入      │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  DeepSeek API    │  ← ChatOpenAI (temperature=0.0)
│  裁判模型调用      │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  JSON 解析与清洗  │  ← 去 think 标签 + JSON 提取
│  异常兜底处理      │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│  评估结果渲染     │  ← Streamlit: Metric / DataFrame / 徽章
│  前端可视化       │
└─────────────────┘
```

## 3. 影响研发的因素分析

### 3.1 技术因素

| 影响因素 | 影响程度 | 缓解措施 |
|---|---|---|
| DeepSeek API 可用性 | 🔴 高 | 备用 API Key；本地缓存测试结果 |
| API 调用速率限制（QPS） | 🟡 中 | 批量评估时增加间隔；staggered requests |
| LLM 输出格式不稳定 | 🔴 高 | 多层 JSON 解析兜底；Prompt 强化约束 |
| Prompt 工程迭代次数 | 🟡 中 | 预留 2-3 轮调优时间；A/B 测试不同措辞 |
| 大文本 Token 超限 | 🟡 中 | 实现文本截断策略；分段评估方案 |
| LangChain 版本兼容性 | 🟢 低 | 锁定依赖版本；requirements.txt 精确版本号 |
| Streamlit 前端渲染性能 | 🟢 低 | 缓存静态资源；分块加载大数据 |

### 3.2 非技术因素

| 影响因素 | 影响程度 | 缓解措施 |
|---|---|---|
| 人工标注数据获取 | 🔴 高 | 优先使用 AI 生成 mock 数据；请同学/老师协助 |
| 评分标准主观性 | 🟡 中 | 制定明确评分细则；多人交叉校验 |
| 课题验收标准变化 | 🟡 中 | 与老师保持每周沟通；模块化设计便于调整 |
| 个人时间管理 | 🔴 高 | 预留 30%~50% 缓冲；整块时间专注编码 |
| 答辩时间节点压缩 | 🟡 中 | 优先完成核心展示功能；非核心功能做降级方案 |

### 3.3 环境因素

| 影响因素 | 影响程度 | 缓解措施 |
|---|---|---|
| 网络访问 GitHub/DeepSeek API | 🔴 高 | 配置代理；本地离线开发 |
| Python 版本兼容性 (3.13+) | 🟡 中 | 使用 venv 隔离环境；测试前确认 Python 版本 |
| Windows/Linux 路径差异 | 🟢 低 | 统一使用 `pathlib.Path` |

### 5.2 计划更新触发条件

| 条件 | 措施 |
|---|---|
| **实际进度超前 ≥ 20%** | 更新计划，将后续 Phase 提前；与老师沟通是否有额外加分需求可加入 |
| **实际进度延迟 ≥ 30%** | 立即更新计划；识别瓶颈任务；考虑降级方案（非核心功能裁剪） |
| **关键阻塞问题 2 天内未解** | 向老师/同学求助；使用备选方案 |
| **验收标准变更** | 重新评估各 Phase 工时；优先保证核心指标达标 |
| **每周五** | 定期检查进度，对比计划 vs 实际，生成周报 |
